Create `.claude/skills/feedback-ingest/SKILL.md` following the established SKILL.md pattern from `fluid-social/SKILL.md`.

**Frontmatter:**
```yaml
name: feedback-ingest
description: "Analyze completed canvas sessions and manual feedback files. Propose and interactively apply updates to brand rules, templates, and skills."
invoke: slash
context: fork
allowed-tools: Bash, Read, Write, Edit, AskUserQuestion
```

**Skill body — the prompt for Claude when /feedback-ingest is invoked:**

1. **Run the engine:** Execute `node tools/feedback-ingest.cjs --dry-run` via Bash. Parse the JSON stdout to get the proposals array and summary stats.

2. **Print summary header:**
   ```
   Feedback Ingestion Report
   Sessions analyzed: {N} new ({M} total)
   Proposals generated: {K}
   Sessions with no signal: {J}
   ```

3. **If 0 proposals:** Print "No actionable patterns found. Run /feedback-ingest again after more sessions accumulate." and stop.

4. **If proposals exist:** Print "Found {K} proposals. Walking through each now." Then for each proposal:
   - Print the proposal summary: type, confidence, target file, scope, evidence summary (session IDs and signal descriptions)
   - Print current text vs proposed text as a diff
   - Use AskUserQuestion: "What would you like to do with this proposal?" with options:
     - `[A] Approve` — accept as-is
     - `[R] Reject` — discard
     - `[M] Modify` — accept with changes (follow-up: "What change would you like to make?")
     - `[S] Skip for now` — defer to next run

5. **Collect ALL decisions before applying any changes.** Never apply partial changes mid-walkthrough (prevents corruption if session times out — see RESEARCH.md Pitfall 4).

6. **After walkthrough summary:**
   ```
   Decisions: Approved: {A}, Rejected: {R}, Modified: {M}, Skipped: {S}
   ```

7. **Apply approved/modified proposals:**
   - For each approved proposal: Read the target file, find the currentText, replace with proposedText (or modified text). Use the Edit tool for surgical changes.
   - After ALL changes applied: run `node tools/compile-rules.cjs` via Bash. If it exits non-zero, warn: "rules.json rebuild had issues — brand docs updated but rules.json may need manual review." Do NOT block on this (known bug, see STATE.md).
   - Run the full engine (without --dry-run) to write the proposal audit trail and update ingested.json: `node tools/feedback-ingest.cjs`
   - Auto-commit: `git add brand/ feedback/ tools/rules.json && git commit -m "brand(learning-loop): apply {N} proposals from {date}"`

8. **If no proposals approved:** Still run the engine (without --dry-run) to update ingested.json and write the audit trail, then print "No changes applied. ingested.json updated."

**Anti-patterns section at the end of the SKILL.md:**
- NEVER auto-apply proposals without walking through each one
- NEVER apply changes one-by-one during the walkthrough — batch all approved changes after collecting all decisions
- NEVER skip the compile-rules.cjs step after applying changes
- NEVER forget to update ingested.json even when no proposals are approved

Create `.claude/skills/fluid-design-os-feedback/SKILL.md` following the established SKILL.md pattern.

**Frontmatter:**
```yaml
name: fluid-design-os-feedback
description: "Capture structured feedback about a generated asset. Guides you through the feedback format and writes a properly formatted file to feedback/."
invoke: slash
context: fork
allowed-tools: Bash, Read, Write, AskUserQuestion
```

**Skill body — the guided feedback capture flow:**

1. **Auto-detect recent session:**
   - List `.fluid/working/` directories matching `/^\d{8}-\d{6}$/` pattern, sorted descending (most recent first)
   - For the most recent session that has a `lineage.json`: read it, extract platform, template, variation count
   - Use AskUserQuestion: "Is this the session you're leaving feedback about?" showing: Session ID, Platform, Template (if any), Number of variations. Options: `[Y] Yes  [N] No, show me other sessions  [M] Manual entry (no session)`

2. **If No — show recent sessions:**
   - List the 5 most recent sessions with lineage.json, showing session ID + platform + date
   - AskUserQuestion: "Which session?" with numbered options + "None of these (manual entry)"

3. **Gather feedback (using AskUserQuestion for each):**
   - "What type of asset?" Options: `brushstroke | circle | logo | font | layout | copy | color` (pre-fill from session platform if available)
   - "What was the outcome?" Options: `[S] Success  [P] Partial — some things worked  [F] Failure — needs rework`
   - "What worked well? (describe briefly, or press Enter to skip)"
   - "What didn't work? What should be different?" (required — this is the main feedback signal)
   - "Any specific rule suggestions? (e.g., 'brushstroke opacity max should be 0.18 not 0.25', or Enter to skip)"

4. **Generate feedback file:**
   - Filename: `feedback/YYYY-MM-DD-{asset_type}-{brief}.md` where brief is a 2-4 word kebab-case slug derived from the "what didn't work" answer (e.g., "opacity-too-high", "text-too-small")
   - YAML frontmatter matching `feedback/README.md` spec:
     ```yaml
     ---
     date: YYYY-MM-DD
     asset_type: {selected type}
     asset_name: {from session lineage or "manual"}
     prompt_used: {from session lineage or "N/A"}
     outcome: {success|partial|failure}
     operator_notes: "{what didn't work answer}"
     rule_weights_affected:
       - rule: "{rule suggestion if provided}"
         current_weight: {look up in tools/rules.json if identifiable, else "unknown"}
         suggested_adjustment: "{the suggestion}"
     ---
     ```
   - Body: combine "what worked" and "what didn't work" as prose, plus session ID reference if applicable

5. **Write file and confirm:**
   - Write the feedback file using the Write tool
   - Print: "Feedback saved to {path}. This will be picked up by /feedback-ingest on next run (bypasses 3-session threshold)."

**Anti-patterns:**
- NEVER skip the session auto-detect step — always offer it first
- NEVER write feedback without confirming the session with the user
- NEVER use a generic filename — always include the asset type and a descriptive brief
