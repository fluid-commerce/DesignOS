Create `tools/feedback-ingest.cjs` as a zero-dependency CJS Node.js tool following the project's dual-output convention (JSON stdout, human summary stderr).

**CLI modes:**
- `node tools/feedback-ingest.cjs` — full run (analyze + write proposals to feedback/proposals/YYYY-MM-DD-proposal.md + output JSON to stdout)
- `node tools/feedback-ingest.cjs --dry-run` — analyze and print proposals to stdout without writing files
- `node tools/feedback-ingest.cjs --test` — run self-tests, exit 0 on pass, non-zero on fail

**Core functions to implement:**

1. `discoverSessions(workingDir)` — returns array of valid session IDs. Pattern: list `.fluid/working/`, filter by `/^\d{8}-\d{6}$/`, filter by `lineage.json` existence. Reference: `canvas/src/lib/sessions.ts`.

2. `loadSessionSignals(workingDir, sessionId)` — returns `{ sessionId, platform, template, signals[] }`. Branch on `lineage.rounds` vs `lineage.entries`. For Phase 4: iterate rounds.variations, merge with annotations.json statuses (annotations.json takes precedence). For Phase 2: iterate entries, derive status from annotations.json only. Extract all annotation text (pin and sidebar types). See RESEARCH.md Pattern 1 for signal record shape.

3. `parseFeedbackFiles(feedbackDir)` — returns array of parsed feedback files. Read `feedback/*.md` (not README.md), split on `---` delimiters, parse YAML frontmatter as simple key: value pairs. Handle `rule_weights_affected` as nested array. These are strong signals that bypass 3-session threshold.

4. `loadIngestedManifest(feedbackDir)` — reads `feedback/ingested.json`, returns `{ processedSessions: [] }`. Creates default manifest if missing.

5. `saveIngestedManifest(feedbackDir, manifest)` — writes updated manifest after successful processing.

Create `feedback/proposals/.gitkeep` for the proposals output directory.

**Anti-patterns to avoid:**

- Do NOT parse variation HTML files — metadata only
- Do NOT use any npm dependencies — Node.js built-ins only (fs, path)

  <action>
Add pattern clustering and proposal generation to `tools/feedback-ingest.cjs`:

**Functions to add:**

6. `clusterSignals(allSignals, feedbackEntries)` — groups signals into pattern clusters.
   - Normalize annotation text to extract topic keywords (brushstroke, opacity, circle, font, copy density, web-vs-social feel, text size, color usage)
   - Group by `(asset_type, signal_topic)` tuples
   - Count distinct sessions per group
   - Apply thresholds:
     - Standard patterns: 3+ distinct sessions required
     - Explicit annotations (contains "NEVER", "always", "don't", "stop using", or similar directive language): 1 session is enough
     - feedback/*.md entries: always qualify regardless of session count
   - Handle conflicting signals: if a topic has both positive (winner) and negative (rejection) signals with >= 40/60 split, flag as conflicting

7. `scopeProposal(cluster)` — determines target file.
   - If cluster signals come from only one asset type (e.g., all social): target the asset-specific doc (`brand/social-post-specs.md`, `brand/website-section-specs.md`)
   - If cluster signals span 2+ asset types: target brand-level doc (`brand/design-tokens.md`, `brand/voice-rules.md` — pick based on topic: color/spacing/opacity -> design-tokens, copy/voice/messaging -> voice-rules)

8. `generateProposals(clusters)` — produces proposal objects.
   - Four types: `weight-adjustment`, `rule-modification`, `new-rule`, `template-update`
   - Each proposal: `{ type, confidence: 'HIGH'|'MEDIUM'|'LOW', target, scope: 'asset-specific'|'brand-level', currentText, proposedText, evidence: [{ sessionId, signal }], conflicting: boolean }`
   - Confidence: HIGH = 5+ sessions or explicit feedback file, MEDIUM = 3-4 sessions, LOW = threshold-bypass single annotation

9. `writeProposalFile(proposals, outputDir)` — writes `feedback/proposals/YYYY-MM-DD-proposal.md` per RESEARCH.md Pattern 4 format: header with summary stats, then each proposal as a numbered section with type, confidence, target, scope, current text, proposed text, and evidence list.

10. `main()` — orchestrates full run:
    - Discover sessions, diff against ingested.json
    - Load signals for new sessions + parse feedback files
    - Cluster signals, generate proposals
    - Write proposal file (unless --dry-run)
    - Update ingested.json manifest
    - Output JSON to stdout (proposals array), summary to stderr
    - Always print summary even with 0 proposals: "Analyzed N new sessions. Found M patterns with proposals. K sessions had no actionable signal."
