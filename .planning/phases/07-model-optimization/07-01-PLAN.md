# Phase 07-01: Model Optimization — Safe Downgrades

## Goal
Assign optimal model tiers to each pipeline agent, reducing cost and latency without quality regression.

## Summary of Changes

| Agent | Before | After | Change |
|-------|--------|-------|--------|
| layout-agent | `model: sonnet` | `model: haiku` | Frontmatter change |
| copy-agent | `model: sonnet` | `model: sonnet` | No change (confirmed correct) |
| styling-agent | `model: sonnet` | `model: sonnet` | No change (confirmed correct) |
| spec-check-agent | `model: sonnet` | `model: sonnet` | No change (confirmed correct) |
| Orchestrator Agent calls | no `model` param | explicit `model` param per agent | All 3 orchestrator skills updated |

## Tasks

### Task 1: Downgrade layout-agent to Haiku

**File:** `.claude/agents/layout-agent.md`
**Change:** `model: sonnet` → `model: haiku`

**Rationale:** Layout agent does template matching — reads archetype from copy.md, selects from ~6 known layout patterns, outputs structural HTML with SLOT comments. No creative judgment. Well-documented output format. Spec-check catches any errors.

### Task 2: Add explicit model overrides in orchestrator Agent calls

Update all three orchestrator slash commands to pass explicit `model` parameters when delegating to subagents. This makes model selection deliberate and visible.

**Files:**
- `~/.claude/commands/fluid-social.md` (also update source if it lives in `.agents/skills/`)
- `~/.claude/commands/fluid-one-pager.md`
- `~/.claude/commands/fluid-theme-section.md`

**Pattern — update delegation instructions from:**
```
Delegate to `copy-agent` via the Agent tool:
```

**To:**
```
Delegate to `copy-agent` via the Agent tool with `model: "sonnet"`:
```

**Model assignments per agent in orchestrator delegations:**
- copy-agent: `model: "sonnet"`
- layout-agent: `model: "haiku"`
- styling-agent: `model: "sonnet"`
- spec-check-agent: `model: "sonnet"`

Add these to BOTH the initial pipeline run AND fix loop re-delegations.

### Task 3: Update orchestrator status output

Add model info to the run header so the operator can see which models are being used:

```
Generating Fluid social post...
  Platform: instagram (1080x1080)
  Product: connect
  Template: (none)
  Models: copy=sonnet, layout=haiku, styling=sonnet, spec-check=sonnet
```

### Task 4: Validation — run test generations

Run 3 test generations to verify no quality regression:

1. `/fluid-social "3AM server fire payment retry story" --platform instagram --debug`
2. `/fluid-one-pager "Fluid Connect product overview" --type product-feature --debug`
3. `/fluid-theme-section "hero section with pain-point headline" --type hero --debug`

**Success criteria:**
- All 3 pass spec-check (or pass within 1 fix iteration)
- Layout HTML contains all expected SLOT comments
- No new failure modes introduced
- Pipeline completes faster than baseline (subjective, note timing)

### Task 5: Document model strategy

Add a `MODEL-STRATEGY.md` to `.planning/` that documents the rationale for each model assignment so future phases can reference it.

## Rollback Plan

If Haiku layout-agent causes quality issues:
1. Revert `layout-agent.md` frontmatter to `model: sonnet`
2. Remove explicit `model: "haiku"` from orchestrator delegation instructions
3. No other changes needed — all other agents stay at Sonnet

## Dependencies

- None — this phase is independent of all prior phases
- Changes are additive (model parameter) or single-line edits (frontmatter)
