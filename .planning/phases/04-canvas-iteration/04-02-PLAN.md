Build the MCP stdio server that agents use to interact with the canvas: pushing assets, reading annotations, checking variation statuses, accessing revision history, and receiving iteration requests.

Purpose: This is the agent-to-canvas bridge. Without it, agents can only write files and hope the canvas picks them up. With it, agents get structured access to annotations, statuses, and iteration context -- enabling the full feedback loop.
Output: Standalone MCP server process in canvas/mcp/ that registers 5 tools, communicable via stdio transport.

Key research findings:
- MCP server MUST be a separate stdio process (cannot embed in Vite dev server)
- Use @modelcontextprotocol/sdk with McpServer + StdioServerTransport
- Use zod for input schema validation
- NEVER use console.log() in MCP server code (corrupts stdio protocol) -- use console.error() only
- Server reads/writes .fluid/working/ directory -- same directory canvas watches
- Iteration requests use a file-based mechanism: canvas writes iterate-request.json, agent reads via MCP tool

Types from Plan 01's canvas/src/lib/types.ts will be used, but the MCP server has its own tsconfig and can duplicate the needed interfaces in canvas/mcp/types.ts to avoid cross-compilation issues.
</context>

  <action>
Create the MCP server as a standalone Node.js process in canvas/mcp/:

1. **MCP tsconfig (canvas/mcp/tsconfig.json):**
   - Target ES2022, module NodeNext, moduleResolution NodeNext
   - outDir: ./dist
   - Include: ./*.ts, ./tools/*.ts
   - Separate from the React app's tsconfig

2. **MCP types (canvas/mcp/types.ts):**
   - Duplicate the annotation and lineage types from the main app types (to avoid cross-project TypeScript complexity)
   - Add MCP-specific types: IterateRequest, PushAssetInput, etc.

3. **MCP server entry (canvas/mcp/server.ts):**
   - Import McpServer from "@modelcontextprotocol/sdk/server/mcp.js"
   - Import StdioServerTransport from "@modelcontextprotocol/sdk/server/stdio.js"
   - Create server with name "fluid-canvas", version "1.0.0"
   - Register all 5 tools (imported from tools/)
   - Connect to StdioServerTransport
   - CRITICAL: Use console.error() for ALL logging -- never console.log()
   - Resolve WORKING_DIR as path.resolve(__dirname, '../../.fluid/working') (relative to canvas/mcp/)

4. **Tool: push_asset (canvas/mcp/tools/push-asset.ts):**
   - Input: sessionId (string), variationId (string), html (string), platform (string, optional)
   - Creates .fluid/working/{sessionId}/{variationId}/styled.html with the HTML content
   - Creates or updates lineage.json with the new variation entry
   - Returns success confirmation with file path
   - Use zod for input validation

5. **Tool: read_annotations (canvas/mcp/tools/read-annotations.ts):**
   - Input: sessionId (string)
   - Reads .fluid/working/{sessionId}/annotations.json
   - Returns full annotations array, or empty array if file doesn't exist
   - Include variation statuses in response

6. **Tool: read_statuses (canvas/mcp/tools/read-statuses.ts):**
   - Input: sessionId (string)
   - Reads annotations.json and extracts just the statuses map
   - Returns: Record<variationPath, VariationStatus>
   - Useful for agent to quickly check which variation won

7. **Tool: read_history (canvas/mcp/tools/read-history.ts):**
   - Input: sessionId (string)
   - Reads lineage.json and annotations.json
   - Returns combined revision history: all rounds, all variations with statuses, all annotations
   - This is the tool agents use to avoid repeating rejected patterns
   - Handle both Phase 2 (entries[]) and Phase 4 (rounds[]) lineage formats

8. **Tool: read_iteration_request (canvas/mcp/tools/iterate.ts):**
   - Input: sessionId (string)
   - Reads .fluid/working/{sessionId}/iterate-request.json if it exists
   - Returns the iteration request (feedback text, winner path, context bundle) or null
   - After reading, renames the file to iterate-request-read.json (so it's not re-read)
   - This is how the canvas UI sends iteration instructions to the agent

9. **Add build script to canvas/package.json:**
   - Add `"build:mcp": "npx tsc -p mcp/tsconfig.json"` to scripts
   - Add `"mcp:dev": "npx tsx mcp/server.ts"` for development (tsx runs TypeScript directly)
   - Install tsx as devDependency if not already present

10. **Add .mcp.json to project root** (or document the `claude mcp add` command):
    ```json
    {
      "mcpServers": {
        "fluid-canvas": {
          "type": "stdio",
          "command": "npx",
          "args": ["tsx", "canvas/mcp/server.ts"]
        }
      }
    }
    ```
