<objective>
Add the annotation system (spatial pins + sidebar notes) and the iteration timeline to the canvas. This turns the canvas from a passive viewer into an active review and iteration tool.

Purpose: Annotations let team members give specific, positioned feedback on variations. The timeline visualizes the iteration journey and enforces the "fan out -> pick winner -> refine" workflow. Together, they create the structured review loop that agents read via MCP.
Output: Fully interactive canvas with pin annotations, sidebar notes, variation status management, branching timeline, and iteration triggering.
</objective>

Key patterns from research:
- Pin positions stored as percentages (0-100) of native asset dimensions to handle iframe scaling
- Click coordinates conversion: pinX = (clickX / displayWidth) * 100, pinY = (clickY / displayHeight) * 100
- annotations.json sits alongside assets in each session directory
- Branching tree shape: one in -> fan out -> pick one -> one in -> fan out -> pick one -> final
- Canvas enforces winner selection before next round
- Iterate button bundles full context (winner, annotations, rejection history) into iterate-request.json

Types defined in Plan 01's canvas/src/lib/types.ts: Annotation, AnnotationReply, AnnotationFile, VariationStatus, Round, etc.

Build the annotation system (store, hooks, and annotation-specific components -- AssetFrame wiring handled in Task 2):

1. **Annotation store (canvas/src/store/annotations.ts):**
   - State: `annotations: Annotation[]`, `statuses: Record<string, VariationStatus>`, `activePin: string | null`
   - Actions:
     - `loadAnnotations(sessionId)`: Fetches from /api/annotations/{sessionId}, populates state
     - `addAnnotation(annotation)`: Adds to state and persists to API
     - `addReply(annotationId, reply)`: Adds reply to existing annotation
     - `setStatus(variationPath, status)`: Sets winner/rejected/final/unmarked
     - `saveAnnotations(sessionId)`: Persists current state to /api/annotations/{sessionId}
     - `setActivePin(id | null)`: For showing/hiding thread panels
   - Auto-save: debounced save after every mutation (300ms)

2. **Annotation persistence hook (canvas/src/hooks/useAnnotations.ts):**
   - Loads annotations when activeSessionId changes
   - Provides convenience methods wrapping the store
   - Exposes `annotationsForVariation(path)` filtered getter

3. **Add API endpoints to Vite server (update canvas/src/server/watcher.ts or add api.ts):**
   - GET /api/annotations/:sessionId -> reads annotations.json, returns AnnotationFile
   - POST /api/annotations/:sessionId -> writes annotations.json
   - These are simple JSON file read/write endpoints

4. **AnnotationPin component:**
   - Renders a numbered circle (1, 2, 3...) at the pin position
   - On click: toggles the annotation thread panel
   - Visually: small colored circle with number, subtle shadow
   - Different colors for human (blue) vs agent (purple) annotations

5. **AnnotationThread component:**
   - Popover or panel that appears when a pin is clicked
   - Shows the original annotation text, author, timestamp
   - Shows replies in chronological order
   - Input field to add a reply
   - Close button

6. **SidebarNotes component:**
   - Right sidebar panel (toggleable) showing all sidebar-type annotations for the active session
   - Grouped by variation
   - Add note button: text input + variation selector
   - Each note shows author, authorType badge, timestamp, text

7. **Annotation tests (canvas/src/__tests__/annotations.test.ts):**
   - Test: Adding a pin annotation stores correct x/y percentages
   - Test: Adding a reply appends to the correct annotation
   - Test: Setting status updates the statuses map
   - Test: Annotations persist and reload correctly (mock API)
   - Test: Pin positions render at correct percentage coordinates

Wire annotations into AssetFrame and build the iteration timeline:

1. **Update AssetFrame component (bridge annotations into existing UI):**
   - Enable pointer-events on the overlay div (was pointer-events: none in Plan 01)
   - On click: calculate pin position as percentage coordinates
   - Show a prompt/input for annotation text
   - Call `addAnnotation()` with type: 'pin', x/y percentages, text
   - Render existing pin markers as numbered circles at their percentage positions
   - Pin markers use CSS `left: ${x}%`, `top: ${y}%` with absolute positioning on the overlay
   - Each variation frame has a status dropdown or click-to-cycle button
   - Statuses: unmarked (default) -> winner -> rejected -> final

2. **Timeline component (canvas/src/components/Timeline.tsx):**
   - Reads rounds from lineage.json (via session data in store)
   - Renders a vertical timeline showing the iteration journey
   - Each round is a TimelineNode
   - Visual shape: single node -> fan-out lines to variations -> converge on winner -> next single node
   - Active round (latest) is expanded, older rounds are collapsed
   - If no rounds data (Phase 2 format), show a single node with the session entries

3. **TimelineNode component:**
   - Shows round number and timestamp
   - Shows variations as small thumbnails or labels with StatusBadge
   - Winner variation is highlighted (green border, checkmark)
   - Rejected variations are dimmed (reduced opacity, strikethrough label)
   - Clicking a variation in the timeline scrolls/selects it in the main VariationGrid
   - PromptReveal shows the prompt used for this round

4. **PromptReveal component:**
   - Collapsible section showing the prompt/refinement instruction for a round
   - Collapsed by default, click to expand
   - Shows the full prompt text, formatted

5. **Status management logic:**
   - When "winner" is set on one variation, all others in the same round auto-set to "rejected" (unless already "final")
   - Only one winner per round
   - Canvas enforces: cannot trigger a new iteration round if no winner is marked for the current round

6. **IteratePanel component:**
   - Appears below the variation grid when a winner is selected
   - Text area for feedback/refinement instructions
   - "Iterate" button
   - On click: calls context-bundler to create iterate-request.json
   - Disabled state when no winner is marked (with helper text: "Mark a winner to iterate")

7. **Context bundler (canvas/src/lib/context-bundler.ts):**
   - `bundleContext(sessionId, feedback)`: Creates an iterate-request.json file containing:
     - sessionId
     - winnerPath (path to the winner variation)
     - feedback (user's text)
     - roundNumber (next round number)
     - rejectedPatterns: array of annotation texts from rejected variations (so agent knows what to avoid)
     - allAnnotations: all annotations for context
     - timestamp
   - Writes to .fluid/working/{sessionId}/iterate-request.json via POST /api/iterate

8. **Add API endpoint:**
   - POST /api/iterate/:sessionId -> writes iterate-request.json to the session directory

9. **Update App.tsx layout:**

   - Add IteratePanel below VariationGrid
   - Add toggleable SidebarNotes panel (right side)
   - Layout: left=SessionSidebar | center=VariationGrid+IteratePanel | right=Timeline+SidebarNotes

    - Test: Setting a winner auto-rejects other variations in the same round
