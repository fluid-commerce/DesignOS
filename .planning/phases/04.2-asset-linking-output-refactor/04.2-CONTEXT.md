**Gathered:** 2026-03-16

<domain>
## Phase Boundary

Replace inline base64-encoded images/fonts in generated HTML with URL-linked assets served via Vite. Establish a DB-backed asset catalog (brand_assets table) with auto-scan population, API-based agent discovery, and per-campaign asset storage. Reduce variation file sizes from 620KB+ to ~10-50KB. </domain>

<decisions>
## Implementation Decisions

### Asset Storage Model
- DB catalog + disk files: SQLite `brand_assets` table tracks metadata (name, category, file_path, mime_type, size_bytes, tags), actual files stay on disk in `assets/`
- Vite continues serving files from disk via existing `/fluid-assets/` middleware
- No BLOB storage — files are too large (67MB total), would bloat DB and slow queries

### Asset Catalog Population
- Auto-scan `assets/` directory on startup, seed DB rows from folder structure (subfolder = category)
- Manual enrichment (tags, descriptions) available later through the Assets page UI
- If a new file appears in `assets/`, next startup picks it up automatically

### Agent Asset Discovery
- API endpoint: `GET /api/assets?category=brushstrokes` returns available assets with `/fluid-assets/` URLs
- Agents call this at generation time to discover what's available — no hardcoded paths in skill files
- MCP tool access deferred (can add later if needed for Claude Code/Cursor agents)

### Agent Asset Referencing in HTML
- Absolute `/fluid-assets/` paths in generated HTML: `src="/fluid-assets/brushstrokes/brush-orange.png"`
- Fonts via `@font-face` with `url('/fluid-assets/fonts/flfontbold.ttf')` — no more base64 data URIs
- Simple, works with existing middleware, no post-processing needed

### Session / Campaign Assets
- Campaign-specific images (uploaded photos, generated graphics) stored at `.fluid/campaigns/{cId}/assets/`
- DB record in `campaign_assets` table linking image to its campaign (queryable, UI can show what a campaign uses)
- Brand assets managed through the Assets page in the UI — separate from campaign assets
- Two clear buckets: reusable brand assets (assets/) vs campaign-specific (`.fluid/campaigns/{cId}/assets/`)

### Export
- ZIP bundle for HTML exports: HTML file + assets/ folder with referenced files
- Replaces current ?download=1 base64 conversion approach
- HTML + assets only — no rendered screenshots (JPG/WebP export stays separate)

### Directory Fix
- Renamed `04.2-install-process-safety/` to `04.3-install-process-safety/` to match roadmap
- Created `04.2-asset-linking-output-refactor/` for this phase

### Claude's Discretion
- DB schema details (indexes, constraints, column types beyond what's specified)
- Asset scan implementation (startup hook vs CLI command vs migration)
- `/fluid-campaigns/` middleware for serving campaign-specific assets
- CreationFrame srcDoc-to-src migration approach
- Agent skill prompt update wording

</decisions>
