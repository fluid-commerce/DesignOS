## Components Changed This Phase

This section describes the interaction contracts for each UI component that changes. No new page-level layouts are introduced.

### 1. ExportActions — HTML button changes to ZIP download

**Current behavior:** "Download HTML" fetches `?download=1`, receives base64-inlined HTML, downloads a `.html` file.

**New behavior:** "Download HTML" changes label to "Download ZIP". Fetches `?export=zip`, receives a ZIP file. Downloads as `.zip`.

**States to handle:**
| State | Visual |
|-------|--------|
| Idle | Button reads "Download ZIP" |
| Loading | Button reads `...`, `disabled`, opacity 0.6 (same as other export buttons) |
| Error | Red error text below button row (`color: #f87171`, `fontSize: 0.7rem`) |

**No layout changes.** Button dimensions, spacing, and typography are identical to existing JPG/WebP buttons. Only the label and fetch URL change.

**Error copy:** `Export failed: {error description}` — matches existing error pattern.

### 2. CreationFrame — srcDoc to src switch

**Current behavior:** Accepts `html: string` prop, renders `<iframe srcDoc={html} sandbox="allow-same-origin" />`.

**New behavior:** Accepts `iterationId: string` prop (replaces `html: string`). Renders `<iframe src={/api/iterations/${iterationId}/html} sandbox="allow-same-origin allow-scripts" />`.

**Visible behavior change:** None — the iframe still renders the same asset HTML. The only visible difference is that the "Preview too large" guard changes:

| State | Visual |
|-------|--------|
| Normal | iframe renders asset at scaled dimensions — no change |
| Loading (new) | Iframe shows browser's built-in loading state briefly (no spinner needed — duration is negligible on localhost) |
| Error (new) | Iframe's built-in `about:blank` fallback — preserve existing empty-state pattern below |

**Removed guard:** The `htmlTooLarge` check (`html.length > 10_000_000`) is removed because `src` mode never holds HTML in memory. If the endpoint 404s, the iframe is blank — acceptable behavior, no special UI needed.

**Props interface change (breaking):**
```
BEFORE: { html: string; name: string; path: string; ... }
AFTER:  { iterationId: string; name: string; path: string; ... }
```

All call sites (`VersionGrid`, any other consumer) must pass `iterationId` instead of `html`. TypeScript enforces this.

**Annotation overlay:** Pin placement, star toggle, and annotation thread popover are **unchanged** — they are on the overlay `div` above the iframe, not inside it.

**sandbox attribute:** Change from `"allow-same-origin"` to `"allow-same-origin allow-scripts"`. Required for injected scripts (postMessage listener, user state injection) to execute in `src`-mode iframes.

### 3. AssetsScreen (future) — brand_assets catalog display

This phase does **not** change AssetsScreen UI. The `GET /api/brand-assets` endpoint is infrastructure-only — AssetsScreen UI remains unchanged and continues to use `GET /api/assets` (saved_assets). Any Assets page UI for browsing brand_assets is deferred per CONTEXT.md.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (export) | "Download ZIP" (replaces "Download HTML") |
| Empty state (iframe, no iteration) | "No preview available" (preserve existing pattern) |
| Error state (export fails) | "Export failed: {error description}" — matches existing ExportActions error pattern |
| Error state (preview too large — removed) | Guard removed; no replacement copy needed |
| Destructive confirmation | None — no destructive actions in this phase |

**Anti-pattern note for agents (skill prompt updates):**
```
NEVER embed assets as base64 data URIs. Always reference via /fluid-assets/ URL paths.
The canvas serves assets via Vite middleware. Base64 blobs create 2-3MB HTML files
that crash the canvas when 6+ variations are displayed simultaneously.
```
This copy appears in updated fluid-social SKILL.md and fluid-one-pager SKILL.md anti-patterns sections.
