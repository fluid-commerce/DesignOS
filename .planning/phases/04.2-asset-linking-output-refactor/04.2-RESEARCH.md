# Phase 04.2: Asset Linking & Output Refactor - Research

**Researched:** 2026-03-16
**Domain:** Vite middleware, SQLite schema extension, React iframe rendering, ZIP export, agent prompt engineering
**Confidence:** HIGH

## Summary

Phase 04.2 replaces inline base64-encoded images and fonts in generated HTML with URL-linked assets served via the existing `/fluid-assets/` Vite middleware. It establishes a `brand_assets` SQLite table populated by auto-scanning `assets/` at startup, exposes a `GET /api/assets?category=` discovery endpoint for agents, adds a `/fluid-campaigns/` middleware for per-campaign asset serving, replaces the `?download=1` base64-inlining export with a server-side ZIP bundle, and switches `CreationFrame` from `srcDoc` to `src` so iframes can resolve absolute asset URLs.

The codebase already has all the key infrastructure in place: `/fluid-assets/` middleware exists in `watcher.ts` (lines 210-226), JSZip v3.10.1 is already present in `templates/jszip.min.js`, the `?download=1` base64 conversion logic is at lines 864-883 of `watcher.ts`, and better-sqlite3 is in use with a proven migration pattern (`try/catch ALTER TABLE`). This phase is primarily additive: new DB tables, a new API endpoint, a new middleware, updated agent skill prompts, and a component `src` attribute switch.

The two biggest technical risks are: (1) the `CreationFrame` `srcDoc` to `src` switch, which requires a URL-based endpoint instead of inline HTML and will need the annotation/pin overlay interaction model to be preserved; and (2) the ZIP bundle export, which needs to run server-side (Node.js `archiver` or JSZip in Node) since the client-side JSZip from `templates/` is browser-only.

**Primary recommendation:** Build incrementally — Plan 1 covers server-side infrastructure (DB tables, asset scan, API endpoint, campaign asset middleware, ZIP export endpoint), Plan 2 covers client-side changes (CreationFrame src switch, ExportActions ZIP trigger, skill prompt updates).

## Architecture Patterns

### DB Schema Extensions

Two new tables follow the existing pattern in `canvas/src/lib/db.ts`:

```typescript
// Source: canvas/src/lib/db.ts initSchema() pattern
db.exec(`
  CREATE TABLE IF NOT EXISTS brand_assets (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    category TEXT NOT NULL,
    file_path TEXT NOT NULL UNIQUE,
    mime_type TEXT NOT NULL,
    size_bytes INTEGER NOT NULL DEFAULT 0,
    tags TEXT NOT NULL DEFAULT '[]',
    created_at INTEGER NOT NULL
  );

  CREATE TABLE IF NOT EXISTS campaign_assets (
    id TEXT PRIMARY KEY,
    campaign_id TEXT NOT NULL,
    file_path TEXT NOT NULL,
    url_path TEXT NOT NULL,
    mime_type TEXT NOT NULL,
    size_bytes INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (campaign_id) REFERENCES campaigns(id)
  );
`);
```

Key notes:
- `brand_assets.file_path` is UNIQUE — prevents duplicate rows for same file across rescans
- `tags` is JSON array (TEXT column), matching the existing `channels` pattern
- `campaign_assets.url_path` stores the `/fluid-campaigns/{cId}/assets/{filename}` URL for serving
- Both tables use `CREATE TABLE IF NOT EXISTS` so startup scan is idempotent

### Asset Auto-Scan Pattern

The scan runs once at Vite plugin startup (`configureServer` hook) before any requests. It mirrors the existing file-watching pattern but is a one-shot scan, not a watcher.

```typescript
// Pattern: scan on startup, INSERT OR IGNORE (upsert by file_path)
async function scanAndSeedBrandAssets(assetsDir: string): Promise<void> {
  const db = getDb();
  const insertStmt = db.prepare(`
    INSERT OR IGNORE INTO brand_assets
      (id, name, category, file_path, mime_type, size_bytes, tags, created_at)
    VALUES (?, ?, ?, ?, ?, ?, ?, ?)
  `);

  const categories = await fs.readdir(assetsDir, { withFileTypes: true });
  for (const entry of categories) {
    if (!entry.isDirectory()) continue;
    const category = entry.name;  // folder name = category
    const categoryDir = path.join(assetsDir, category);
    const files = await fs.readdir(categoryDir);
    for (const file of files) {
      const filePath = path.join(category, file);  // relative: "brushstrokes/brush-texture-01.png"
      const fullPath = path.join(categoryDir, file);
      const stat = await fs.stat(fullPath);
      const ext = path.extname(file).toLowerCase();
      const mime = MIME_TYPES[ext] || 'application/octet-stream';
      insertStmt.run(nanoid(), file.replace(/\.[^.]+$/, ''), category, filePath, mime, stat.size, '[]', Date.now());
    }
  }
}
```

`INSERT OR IGNORE` ensures idempotency — running on every startup won't create duplicates. Files deleted from disk will still have DB rows (acceptable for now; manual cleanup available via Assets page).

### GET /api/assets Endpoint Pattern

Follows the existing API middleware pattern in watcher.ts exactly:

```typescript
// Pattern matches: GET /api/assets or GET /api/assets?category=brushstrokes
if (url === '/api/assets' || url.startsWith('/api/assets?')) {
  const category = new URL(req.url!, 'http://localhost').searchParams.get('category');
  const db = getDb();
  const rows = category
    ? db.prepare('SELECT * FROM brand_assets WHERE category = ? ORDER BY name ASC').all(category)
    : db.prepare('SELECT * FROM brand_assets ORDER BY category ASC, name ASC').all();

  const assets = rows.map((row: any) => ({
    id: row.id,
    name: row.name,
    category: row.category,
    url: `/fluid-assets/${row.file_path}`,   // absolute URL for agent use
    mimeType: row.mime_type,
    sizeBytes: row.size_bytes,
    tags: JSON.parse(row.tags),
  }));
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify(assets));
  return;
}
```

The response shape gives agents a ready-to-use `url` field — agents don't need to construct paths.

### CreationFrame: srcDoc to src Switch

This is the most structurally significant client change. `CreationFrame` currently accepts `html: string` and passes it as `srcDoc`. The new version accepts `iterationId: string` and uses `src="/api/iterations/{id}/html"`.

**Why this works:** The `/api/iterations/:id/html` endpoint already exists in watcher.ts (lines 782-945+) and already serves HTML with:
- Asset path rewrites (`../../assets/` → `/fluid-assets/`)
- `<base>` tag injection for relative URL resolution
- User state injection via inline script

**The key change in CreationFrame:**

```tsx
// BEFORE (srcDoc — loads full HTML blob into React component state)
<iframe
  srcDoc={html}
  sandbox="allow-same-origin"
  ...
/>

// AFTER (src — browser fetches HTML directly, assets load via separate requests)
<iframe
  src={`/api/iterations/${iterationId}/html`}
  sandbox="allow-same-origin allow-scripts"
  ...
/>
```

**Critical implication:** `sandbox="allow-same-origin"` without `allow-scripts` prevents the `<script>` tags in iteration HTML (like the postMessage listener and user state injection) from running. When switching to `src`, you need `allow-scripts` added. The existing `srcDoc` usage also needs `allow-scripts` for any template HTML with scripts. Check current behavior carefully.

**Props change:** The parent components (wherever `CreationFrame` is used) pass `html` content today. They will need to pass `iterationId` instead. This is a breaking prop change — track all call sites.

**Where CreationFrame is used:** Grep shows it's a component used in the session-based variation display flow. The `DrillDownGrid` and campaign detail views use `AppShell` iframes differently (they use the `/api/iterations/:id/html` URL directly already). `CreationFrame` is specifically the old-style session canvas view.

### ZIP Export Pattern

Server-side ZIP endpoint replaces the `?download=1` query parameter logic. New endpoint: `GET /api/iterations/:id/html?export=zip`.

```typescript
// Server-side using archiver
import archiver from 'archiver';

if (isZipExport) {
  const archive = archiver('zip', { zlib: { level: 6 } });

  res.writeHead(200, {
    'Content-Type': 'application/zip',
    'Content-Disposition': `attachment; filename="fluid-asset-${iterationId}.zip"`,
  });

  archive.pipe(res);

  // 1. Add the HTML file (with /fluid-assets/ paths intact — they work when opened via server)
  archive.append(html, { name: `${iterationId}/index.html` });

  // 2. Find all /fluid-assets/ references and add them
  const assetRegex = /\/fluid-assets\/([^"'\s]+)/g;
  let match;
  while ((match = assetRegex.exec(html)) !== null) {
    const assetRelPath = match[1];
    const fullPath = path.join(assetsDir, assetRelPath);
    try {
      archive.file(fullPath, { name: `${iterationId}/assets/${assetRelPath}` });
    } catch { /* skip missing */ }
  }

  archive.finalize();
  return;
}
```

**Alternative (no new dependency):** Use Node's built-in `zlib` + manually write a ZIP. This is complex and error-prone. Recommend `archiver`.

**ExportActions change:** Replace the `fetch('/api/iterations/${iteration.id}/html?download=1')` call with `fetch('/api/iterations/${iteration.id}/html?export=zip')` and change the blob MIME type and filename extension to `.zip`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Server-side ZIP generation | Custom ZIP byte writer | `archiver` npm package | ZIP format has edge cases (CRC, compression levels, directory entries, streaming); archiver handles all of this and streams to res |
| DB idempotency on rescan | Custom "check then insert" logic | `INSERT OR IGNORE` (SQLite native) | Single SQL statement, atomic, no race conditions |
| Content-type for assets | Custom lookup map | Extend existing `MIME_TYPES` in watcher.ts (line 1510-1528) | Map already covers all needed types: .png, .jpg, .svg, .ttf, .woff2 |
| Asset path extraction from HTML for ZIP | Custom HTML parser | Regex on `/fluid-assets/` pattern | HTML is known structure; full parser is overkill for URL extraction |

**Key insight:** The MIME_TYPES map and `serveFluidAsset()` helper in watcher.ts are already the canonical content-type resolver. All new file-serving code must use them.

## Common Pitfalls

### Pitfall 1: sandbox="allow-same-origin" blocks scripts in src-mode iframes

**What goes wrong:** When `CreationFrame` switches from `srcDoc` to `src`, the `<script>` tags in iteration HTML (postMessage listener, user state injection) will be blocked by `sandbox="allow-same-origin"` without `allow-scripts`. Template HTML that currently relies on scripts to render (brushstroke animations, slot value injection) will silently break.

**Why it happens:** `srcDoc` iframes have different sandbox behavior from `src` iframes. With `srcDoc`, same-origin content with `allow-same-origin` can run scripts via inline content. With `src` fetching from the same origin, `allow-scripts` is required for any `<script>` tag execution.

**How to avoid:** Update `sandbox` to `"allow-same-origin allow-scripts"` when switching to `src`. Verify the postMessage listener script (injected by watcher.ts line 888-944) still fires after the switch.

**Warning signs:** Template slots not rendering their saved values; postMessage capture for JPG export silently timing out.

### Pitfall 2: CreationFrame call sites — prop breaking change

**What goes wrong:** `CreationFrame` currently accepts `html: string` and `path: string`. After the refactor it needs `iterationId: string`. Any component passing raw HTML to `CreationFrame` will break at compile time (TypeScript catches it) but there may be test mocks that pass HTML strings directly.

**Why it happens:** The session-based canvas view (`VersionGrid`, `SessionSidebar` area) passes HTML content fetched from the server. With the src-switch, those components should instead pass the iteration ID and let the iframe fetch directly.

**How to avoid:** Update `CreationFrameProps` interface first, then fix TypeScript errors at every call site. Run `vitest` after to catch test mocks.

**Warning signs:** TypeScript errors after CreationFrame interface change; test failures in `VersionGrid.test.tsx`.

### Pitfall 3: INSERT OR IGNORE needs UNIQUE constraint to work

**What goes wrong:** `INSERT OR IGNORE` silently does nothing when there's a conflict — but only if a UNIQUE constraint exists on the column. Without `UNIQUE` on `file_path`, it will insert duplicate rows on every startup.

**Why it happens:** SQLite's `INSERT OR IGNORE` suppresses errors from UNIQUE constraint violations specifically. Without the constraint, there's nothing to conflict on.

**How to avoid:** Declare `file_path TEXT NOT NULL UNIQUE` in the `brand_assets` CREATE TABLE statement. Verified this is in the locked schema spec.

**Warning signs:** `brand_assets` table growing with each server restart; `/api/assets` returning duplicate entries.

### Pitfall 4: Campaign assets middleware path traversal risk

**What goes wrong:** `/fluid-campaigns/{cId}/assets/{filename}` could be exploited with `../../` path segments to read arbitrary files from the server.

**Why it happens:** If the middleware blindly concatenates URL segments to `fluidDir`, a path like `/fluid-campaigns/../../../etc/passwd` could succeed.

**How to avoid:** Use the same guard pattern as the patterns middleware (watcher.ts lines 262-263):
```typescript
if (!fullPath.startsWith(path.join(fluidDir, 'campaigns') + path.sep)) {
  return next(); // path traversal attempt — reject
}
```

**Warning signs:** Security audit flag; any error about unexpected file access.

### Pitfall 5: Asset scan at startup delays Vite server readiness

**What goes wrong:** If `assets/` contains hundreds of files (67MB total, 9 categories), synchronous scanning at startup could delay the Vite server from accepting connections.

**Why it happens:** `configureServer` hook in Vite is synchronous in its return but the scan uses `fs.readdir` async. Long async chains before the server is marked ready can time out in some environments.

**How to avoid:** Run the scan as a fire-and-forget async call from `configureServer`. Don't await it in the hook — let requests come in while scan runs in background. The endpoint returns empty results until scan completes (which is acceptable behavior).

**Warning signs:** Vite startup taking >5 seconds; timeout errors in dev startup.

### Pitfall 6: AssetsScreen currently uses /api/assets for saved_assets (not brand_assets)

**What goes wrong:** `AssetsScreen.tsx` already fetches `GET /api/assets` — this currently returns `saved_assets` (user's DAM imports). Adding `brand_assets` to the same endpoint without differentiation will mix the two catalogs.

**Why it happens:** The `GET /api/assets` route in watcher.ts currently returns `getSavedAssets()` data. The new brand asset endpoint needs to be either:
(a) the same path with a `?catalog=brand` param, or
(b) a separate path like `GET /api/brand-assets?category=brushstrokes`

**How to avoid:** CONTEXT.md specifies `GET /api/assets?category=brushstrokes` for brand asset discovery. AssetsScreen uses `GET /api/assets` without a category param. These could conflict. Resolution: use `GET /api/brand-assets?category=` for the new brand asset catalog endpoint, distinct from the existing `saved_assets` endpoint at `GET /api/assets`. This avoids breaking the existing AssetsScreen.

**Warning signs:** AssetsScreen showing brand brushstrokes mixed with user DAM imports; agents receiving saved_assets rows when querying for brand assets.

1. **Should GET /api/brand-assets or GET /api/assets?catalog=brand be the endpoint?**
   - What we know: `AssetsScreen` already uses `GET /api/assets` for `saved_assets`. The CONTEXT.md spec says `GET /api/assets?category=brushstrokes`.
   - What's unclear: Does adding a `category` param to the existing `/api/assets` endpoint accidentally break `AssetsScreen` (which fetches without params)?
   - Recommendation: Use `GET /api/brand-assets?category=` as the new endpoint. Keep `/api/assets` returning `saved_assets` unchanged. This is the safest path. The planner should choose one approach and be explicit.

2. **Does CreationFrame receive an iterationId or a URL?**
   - What we know: The src should be `/api/iterations/{id}/html`. The parent needs to pass enough for this URL.
   - What's unclear: Some call sites may be in the session-based canvas view that doesn't have iteration IDs (older session path). Does the session canvas still use `CreationFrame`?
   - Recommendation: Check usages of `CreationFrame` in `VersionGrid.tsx` and `SessionSidebar.tsx`. If those components have iteration IDs available, the refactor is clean. If not, they may need to be updated to use iteration IDs first.

3. **archiver vs built-in zlib for ZIP?**
   - What we know: `archiver` is the standard Node.js ZIP library, not yet in `package.json`. `templates/jszip.min.js` is browser-only.
   - What's unclear: Whether the team prefers adding a new npm dependency vs writing a minimal ZIP with built-in Node modules.
   - Recommendation: Add `archiver` — it's well-maintained, streaming, and avoids implementing ZIP format internals. The dependency cost is low.
