    **Gap 1 — Mosaic response unwrap (CampaignDashboard.tsx):**

    Two fetch call sites parse the `/api/campaigns/:id/preview-urls` response incorrectly. The endpoint returns `{ urls: PreviewUrl[] }` but the code treats the response as a bare `PreviewUrl[]`.

    Fix 1a — CampaignMosaic component (around line 54):
    Change `return res.json() as Promise<PreviewUrl[]>` to properly unwrap:
    ```typescript
    return res.json().then((data: { urls: PreviewUrl[] }) => data.urls ?? []);
    ```
    Or equivalently, chain the unwrap after the json() call so the downstream `.then()` receives the array.

    Fix 1b — Eager prefetch in CampaignDashboard (around line 620-622):
    Change:
    ```typescript
    .then((urls: PreviewUrl[]) => {
      setMosaicData((prev) => ({ ...prev, [campaign.id]: urls.slice(0, 4) }));
    })
    ```
    To:
    ```typescript
    .then((data: { urls: PreviewUrl[] } | PreviewUrl[]) => {
      const urls = Array.isArray(data) ? data : (data.urls ?? []);
      setMosaicData((prev) => ({ ...prev, [campaign.id]: urls.slice(0, 4) }));
    })
    ```
    Use the defensive `Array.isArray` check so it handles both shapes gracefully.

    Also remove the orphaned `StatusBadge` import from CampaignDashboard.tsx since StatusBadge will be rendered in App.tsx instead (unless it is also used elsewhere in CampaignDashboard — check first and keep if used).

    **Gap 2 — StatusBadge on asset cards (App.tsx):**

    Import `StatusBadge` from `./components/StatusBadge` in App.tsx. In the `assetItems` mapping (around line 153-162), the `genStatus` is already computed but only used in the subtitle string. The DrillDownGrid renders each item via `renderPreview` which returns a `PreviewDescriptor` — the badge needs to be part of the card overlay.

    The approach depends on how DrillDownGrid renders cards. Since DrillDownGrid's `renderPreview` returns a `PreviewDescriptor` (not JSX), the StatusBadge must be added via the `meta.badges` array or as an additional element in the card. Check if DrillDownGrid supports a `renderOverlay` or similar prop. If not, add the StatusBadge to the `meta` fallback path in `renderAssetPreview` (in preview-utils.ts `buildAssetPreview` function) for the non-iframe case.

    For the iframe case (generation complete), the badge is less critical since the iframe itself shows the result. For pending/generating states, the metadata fallback card already shows status in the `badges` array. The key fix is: when genStatus is 'generating' or 'pending', ensure the visual StatusBadge component renders. This can be done by:

    1. Adding a `renderItemExtra` or overlay pattern to the assetItems DrillDownItem — check if DrillDownItem supports a `badge` or `extra` ReactNode field.
    2. If DrillDownGrid doesn't support extra ReactNode overlays, the simplest fix is to use CSS to position a StatusBadge absolutely over the card. Add a wrapper around DrillDownGrid for the campaign view that renders StatusBadge overlays keyed by asset position.
    3. Alternatively, if DrillDownGrid cards have a DOM structure that allows it, inject StatusBadge into the subtitle area of assetItems by making subtitle a ReactNode (check if DrillDownItem.subtitle accepts ReactNode or only string).

    The pragmatic solution: Check DrillDownItem type definition. If `subtitle` accepts `ReactNode`, change the assetItems mapping to include `<StatusBadge status={genStatus} />` inline in the subtitle. If it only accepts string, add StatusBadge rendering via the card's overlay mechanism.
