### Pattern 1: Claude CLI Streaming via SSE

**What:** Spawn `claude -p` from Vite middleware, pipe stdout through SSE to the browser.

**Why SSE over WebSockets:** SSE is simpler (unidirectional server->client), works with standard HTTP, and the user explicitly stated view-only during generation (no client->server messages needed mid-stream). The browser's `EventSource` API handles reconnection automatically.

**Server side (watcher.ts middleware):**
```typescript
// POST /api/generate -- starts generation, returns SSE stream
// Request body: { prompt, template?, customization? }
import { spawn } from 'node:child_process';

if (req.url === '/api/generate' && req.method === 'POST') {
  const body = JSON.parse(await readBody(req));
  const sessionId = generateSessionId(); // YYYYMMDD-HHMMSS format

  // Build the claude command
  const args = [
    '-p', body.prompt,
    '--output-format', 'stream-json',
    '--verbose',
    '--allowedTools', 'Read,Write,Bash,Glob,Grep,Edit,Agent',
  ];

  // CRITICAL: stdin must be 'inherit', not 'pipe' -- piped stdin causes hangs
  // See: https://github.com/anthropics/claude-code/issues/771
  const child = spawn('claude', args, {
    cwd: projectRoot,
    stdio: ['inherit', 'pipe', 'pipe'],
    env: { ...process.env },
  });

  // Set SSE headers
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  });

  // Stream stdout line by line to client
  let buffer = '';
  child.stdout.on('data', (chunk: Buffer) => {
    buffer += chunk.toString();
    const lines = buffer.split('\n');
    buffer = lines.pop() ?? ''; // keep incomplete line in buffer
    for (const line of lines) {
      if (line.trim()) {
        res.write(`data: ${line}\n\n`);
      }
    }
  });

  child.stderr.on('data', (chunk: Buffer) => {
    // Optionally forward stderr as a different event type
    res.write(`event: stderr\ndata: ${JSON.stringify({ text: chunk.toString() })}\n\n`);
  });

  child.on('close', (code) => {
    res.write(`event: done\ndata: ${JSON.stringify({ code, sessionId })}\n\n`);
    res.end();
  });

  // Handle client disconnect
  req.on('close', () => {
    child.kill('SIGTERM');
  });
  return;
}
```

**Client side (useGenerationStream hook):**
```typescript
// Use fetch + ReadableStream instead of EventSource for POST requests
// EventSource only supports GET
export function useGenerationStream() {
  const [events, setEvents] = useState<StreamUIMessage[]>([]);
  const [status, setStatus] = useState<'idle' | 'generating' | 'complete'>('idle');

  const startGeneration = useCallback(async (prompt: string, opts?: GenerateOpts) => {
    setStatus('generating');
    setEvents([]);

    const response = await fetch('/api/generate', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ prompt, ...opts }),
    });

    const reader = response.body!.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;

      buffer += decoder.decode(value, { stream: true });
      const lines = buffer.split('\n\n');
      buffer = lines.pop() ?? '';

      for (const block of lines) {
        const match = block.match(/^(?:event: (\w+)\n)?data: (.+)$/s);
        if (!match) continue;
        const [, eventType, data] = match;
        const parsed = JSON.parse(data);

        if (eventType === 'done') {
          setStatus('complete');
        } else {
          const uiMessage = parseStreamEvent(parsed, eventType);
          if (uiMessage) {
            setEvents(prev => [...prev, uiMessage]);
          }
        }
      }
    }
  }, []);

  return { events, status, startGeneration };
}
```

### Pattern 2: Stream-JSON Event Parsing for Chat UI

**What:** Parse Claude CLI `--output-format stream-json` NDJSON events into renderable chat messages.

The stream-json format emits one JSON object per line. Key event types to handle:

| Event Structure | What It Means | How to Render |
|----------------|---------------|---------------|
| `{ type: "system", ... }` | Session init | Ignore or show as system message |
| `{ type: "stream_event", event: { type: "content_block_start", content_block: { type: "text" } } }` | Text block starting | Prepare new message bubble |
| `{ type: "stream_event", event: { type: "content_block_delta", delta: { type: "text_delta", text: "..." } } }` | Text token | Append to current message bubble |
| `{ type: "stream_event", event: { type: "content_block_start", content_block: { type: "tool_use", name: "Read" } } }` | Tool call starting | Show "[Reading file...]" status |
| `{ type: "stream_event", event: { type: "content_block_stop" } }` | Block finished | Finalize current element |
| `{ type: "assistant", ... }` | Complete turn message | Can use to reconcile/verify accumulated text |
| `{ type: "result", ... }` | Final result | Show completion indicator |

**Parser implementation:**
```typescript
// stream-parser.ts
export interface StreamUIMessage {
  id: string;
  type: 'text' | 'tool-start' | 'tool-done' | 'status' | 'error';
  content: string;
  toolName?: string;
  timestamp: number;
}

export function parseStreamEvent(
  event: any,
  sseEventType?: string
): StreamUIMessage | null {
  if (sseEventType === 'stderr') {
    return { id: nanoid(), type: 'status', content: event.text, timestamp: Date.now() };
  }

  if (event.type === 'stream_event') {
    const inner = event.event;

    if (inner.type === 'content_block_start') {
      if (inner.content_block?.type === 'tool_use') {
        return {
          id: nanoid(),
          type: 'tool-start',
          content: `Using ${inner.content_block.name}...`,
          toolName: inner.content_block.name,
          timestamp: Date.now(),
        };
      }
      return null; // text block start -- wait for deltas
    }

    if (inner.type === 'content_block_delta') {
      if (inner.delta?.type === 'text_delta') {
        return {
          id: nanoid(),
          type: 'text',
          content: inner.delta.text,
          timestamp: Date.now(),
        };
      }
      return null; // input_json_delta for tool args -- skip
    }

    if (inner.type === 'content_block_stop') {
      // Could emit tool-done if we tracked current tool
      return null;
    }
  }

  if (event.type === 'result') {
    return { id: nanoid(), type: 'status', content: 'Generation complete', timestamp: Date.now() };
  }

  return null;
}
```

**Rendering strategy:** Accumulate consecutive text deltas into a single expanding message bubble rather than creating a new bubble per token. Use a ref to batch text deltas and update the last message's content. Tool-start and tool-done events get their own distinct status bubbles.

## Common Pitfalls

### Pitfall 1: Piped stdin Causes Claude CLI to Hang
**What goes wrong:** `claude -p` spawned from Node.js hangs indefinitely with no output.
**Why it happens:** When stdin is set to `'pipe'` (Node.js default), Claude CLI waits for stdin input and never proceeds. This is a documented issue (github.com/anthropics/claude-code/issues/771).
**How to avoid:** Always use `stdio: ['inherit', 'pipe', 'pipe']` -- stdin inherits from parent, stdout and stderr are piped for capture.
**Warning signs:** Child process never emits data events, PID exists but no output.
**Confidence:** HIGH -- verified via official GitHub issue, confirmed resolved with this workaround.

### Pitfall 2: SSE Response Buffering
**What goes wrong:** Stream events arrive in large batches instead of real-time, or the connection appears stuck.
**Why it happens:** Node.js HTTP response and reverse proxies buffer output. Vite's dev server middleware may also buffer.
**How to avoid:** Call `res.flushHeaders()` after `writeHead()`. Disable any compression for the SSE endpoint. Ensure no `Content-Length` header is set (chunked transfer).
**Warning signs:** Events appear in batches of 10-20 instead of one at a time.

### Pitfall 3: Chokidar Not Detecting New Subdirectories
**What goes wrong:** Agent creates a new session directory, chokidar doesn't fire events.
**Why it happens:** Chokidar v4 can miss directories created after watcher starts at depth > 1.
**How to avoid:** Create `.fluid/working/` before starting watcher. Use `depth: 4` and `awaitWriteFinish`. Add periodic re-scan fallback.
**Warning signs:** Sessions only appear after browser refresh.

### Pitfall 4: Output Path Confusion in Skills
**What goes wrong:** Agent writes `styled.html` to `./output/` instead of `.fluid/working/{sessionId}/`.
**Why it happens:** Skills have a dual-path design: working files to `.fluid/working/`, final output copies to `./output/`. Agents sometimes skip the working directory.
**How to avoid:** When canvas drives generation via `claude -p`, the prompt explicitly specifies the output path. For external agents (Desktop fallback), skills check for `.fluid/canvas-active` sentinel file. The prompt sent by the canvas must include explicit output path instructions.
**Warning signs:** Files appear in `output/` but not in canvas.

### Pitfall 5: Context Bundler Requiring Winner for Single Variation
**What goes wrong:** `bundleContext()` throws "No winner selected" even with a single variation.
**Why it happens:** `bundleContext` always requires a winner path. With smart unblock, single variations don't need explicit winner.
**How to avoid:** When there's exactly 1 variation and no explicit winner, auto-infer the winner path in `bundleContext`.

### Pitfall 6: Multiple Concurrent Generations
**What goes wrong:** User starts a second generation while the first is still running. Two `claude -p` processes compete, writing to overlapping directories.
**Why it happens:** No generation lock in the UI.
**How to avoid:** Disable the generate button while a generation is in progress. Track active generation PID server-side and refuse new requests (or kill previous). Each generation gets a unique sessionId, so file conflicts are unlikely, but UI confusion is the risk.

### Pitfall 7: Template HTML Containing Relative Asset Paths
**What goes wrong:** Template previews show broken images/fonts because assets are referenced with relative paths like `../../assets/brushstrokes/brush-1.png`.
**Why it happens:** Templates use relative paths designed for their location in `templates/social/`. When loaded via srcDoc in canvas (at localhost:5174), relative paths don't resolve.
**How to avoid:** The template HTML loaded for preview needs asset paths rewritten to absolute URLs served by the Vite dev server. Add a Vite middleware route like `/template-assets/` that serves the project's `assets/` directory.
**Warning signs:** Template previews show broken images but the HTML structure is correct.

### SSE Stream Connection (client)
```typescript
// Using fetch + ReadableStream for POST-based SSE
async function connectStream(prompt: string, onEvent: (e: StreamUIMessage) => void) {
  const res = await fetch('/api/generate', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ prompt }),
  });

  const reader = res.body!.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });

    // Parse SSE frames
    while (buffer.includes('\n\n')) {
      const idx = buffer.indexOf('\n\n');
      const frame = buffer.slice(0, idx);
      buffer = buffer.slice(idx + 2);

      const dataMatch = frame.match(/^data: (.+)$/m);
      if (dataMatch) {
        const parsed = JSON.parse(dataMatch[1]);
        const msg = parseStreamEvent(parsed);
        if (msg) onEvent(msg);
      }
    }
  }
}
```

### Generate Endpoint with CLI Spawning
```typescript
// POST /api/generate
// Spawns claude -p, streams stdout as SSE
const generateSession = async (req, res, absDir) => {
  const body = JSON.parse(await readBody(req));
  const sessionId = new Date().toISOString().replace(/[-:T]/g, '').slice(0, 15).replace(/(\d{8})(\d{6})/, '$1-$2');
  const sessionDir = path.join(absDir, sessionId);
  await fs.mkdir(sessionDir, { recursive: true });

  const prompt = buildPrompt(body, sessionDir);

  const child = spawn('claude', [
    '-p', prompt,
    '--output-format', 'stream-json',
    '--verbose',
    '--include-partial-messages',
    '--allowedTools', 'Read,Write,Bash,Glob,Grep,Edit,Agent',
  ], {
    cwd: projectRoot,
    stdio: ['inherit', 'pipe', 'pipe'],
    env: { ...process.env },
  });

  // SSE headers
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    'X-Session-Id': sessionId,
  });
  res.flushHeaders();

  // Pipe stdout as SSE data events
  let stdoutBuffer = '';
  child.stdout.on('data', (chunk) => {
    stdoutBuffer += chunk.toString();
    const lines = stdoutBuffer.split('\n');
    stdoutBuffer = lines.pop() ?? '';
    for (const line of lines) {
      if (line.trim()) res.write(`data: ${line}\n\n`);
    }
  });

  child.on('close', (code) => {
    res.write(`event: done\ndata: ${JSON.stringify({ code, sessionId })}\n\n`);
    res.end();
  });

  req.on('close', () => child.kill('SIGTERM'));
};
```

### Template Discovery Endpoint
```typescript
// GET /api/templates
// Reads templates/social/*.html and templates/one-pagers/*.html
const discoverTemplates = async (projectRoot: string) => {
  const categories = [
    { dir: 'templates/social', category: 'social', dims: { width: 1080, height: 1080 } },
    { dir: 'templates/one-pagers', category: 'one-pager', dims: { width: 816, height: 1056 } },
  ];

  const templates: TemplateInfo[] = [];
  for (const { dir, category, dims } of categories) {
    const fullDir = path.join(projectRoot, dir);
    const files = await fs.readdir(fullDir).catch(() => []);
    for (const file of files) {
      if (!file.endsWith('.html') || file === 'index.html') continue;
      const html = await fs.readFile(path.join(fullDir, file), 'utf-8');
      templates.push({
        id: file.replace('.html', ''),
        name: file.replace('.html', '').replace(/-/g, ' '),
        category,
        html,
        dimensions: dims,
      });
    }
  }
  return templates;
};
```

## Open Questions

1. **Claude CLI availability on user's machine**
   - What we know: `claude -p` is available when Claude Code CLI is installed
   - What's unclear: Whether the user always has `claude` in PATH when running canvas standalone
   - Recommendation: Check for `claude` in PATH at startup. If not available, disable generate buttons and show setup instructions. The file watcher fallback still works for Desktop-generated content.

2. **Template asset path resolution in previews**
   - What we know: Templates reference assets with relative paths like `../../assets/`
   - What's unclear: Whether srcDoc iframes can resolve these, or if we need to rewrite paths
   - Recommendation: Add a static asset serving middleware route at `/fluid-assets/` pointing to the project's `assets/` directory. Rewrite template HTML asset paths before injecting into srcDoc. Or, since templates are read server-side, do a simple string replace of relative asset paths to absolute URLs.

3. **Generation prompt construction**
   - What we know: The prompt needs to invoke the right skill (fluid-social, fluid-one-pager) with proper flags
   - What's unclear: Exact prompt format that reliably triggers the right skill pipeline
   - Recommendation: Construct the prompt to explicitly reference the skill: "Use the /fluid-social skill to generate..." This may not work in `-p` mode since skills are interactive-only. Alternative: construct the prompt to replicate what the skill does (specify pipeline steps, output paths, brand docs). The skill SKILL.md content can be appended via `--append-system-prompt-file`.

4. **Skill path sentinel for external agents**
   - What we know: Skills need to know canvas is active to route output correctly
   - What's unclear: Best detection mechanism for Desktop-originated generation
   - Recommendation: Use a `.fluid/canvas-active` sentinel file created by start.sh and removed by stop.sh. Skills check for this file. For canvas-originated generation, the prompt explicitly sets the output path, so the sentinel is only needed for the fallback Desktop workflow.
