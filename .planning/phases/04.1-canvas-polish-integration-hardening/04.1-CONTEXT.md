# Phase 04.1: Canvas Polish & Integration Hardening - Context

**Gathered:** 2026-03-10 (updated 2026-03-11)
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix friction points and harden the canvas + MCP integration built in Phase 4. ALSO: elevate the canvas from a passive viewer into a self-contained creative workspace — the canvas becomes the control plane that drives generation via headless Claude CLI, not a sidecar to Claude Desktop.

Covers: winner/status UX overhaul, canvas-as-control-plane (prompt sidebar + template gallery + headless CLI generation with streaming), file watcher auto-discovery reliability, and skill path hardening.

Does NOT add new canvas features beyond what's described here — this is polish, reliability, and the architectural shift to standalone generation.

</domain>

<decisions>
## Implementation Decisions

### Winner/status UX
- Replace click-to-cycle status with a dedicated star icon for marking winners
- Remove auto-reject behavior — starring a winner leaves other variations unmarked (user can manually reject if desired)
- Smart unblock on iterate panel: with 1 variation, text input AND iterate button are always enabled (no winner pick needed). With 2+ variations, text is always writable but Iterate requires a winner star
- The old status cycle (unmarked->winner->rejected->final) is replaced by: star for winner, separate controls for rejected/final if needed

### Canvas as control plane (NEW)
- **Ditch Claude Desktop as the engine entirely.** Canvas drives ALL generation (initial + iterations) via headless `claude -p` CLI
- Canvas is a self-contained creative workspace — user opens it, generates, iterates, picks winners, all from one UI
- No MCP-to-Desktop triggering. The canvas IS the interface
- **Standalone primary, Desktop fallback:** Canvas generates via its own CLI. If an agent externally writes to `.fluid/working/`, canvas still picks it up via file watcher. But Desktop integration is not the primary workflow

### Prompt sidebar (NEW)
- Left sidebar with a persistent chat-like prompt/conversation panel
- Shows the agent's actual output stream (chat-like, not just status indicators) — user sees which stage the agent is on
- Stream captured via `claude --output-format stream-json` — parse structured events and render as chat messages
- **View-only during generation** — user watches the stream but cannot interject mid-generation. They iterate AFTER results appear via the existing iterate panel
- Sidebar is always present for both "Create with AI" (free prompt) and "Create from template" (template-triggered generation)

### Template gallery (NEW)
- Main pane shows available templates/archetypes as **live HTML previews** (render actual template HTML as thumbnails, like current variation frames)
- Two entry points: "Create from template" (pick template, customize, generate) or "Create with AI" (type a prompt in the sidebar)
- **Template flow:** User picks a template -> quick customization step (headline text, accent color, topic/brief) -> then generate. Not immediate generation — customization step first for more control over first output
- Asset type selection implicit in template choice (social post templates, one-pager templates, etc.)

### File watcher auto-discovery
- Fix auto-discovery so new sessions appear in canvas immediately without browser refresh
- Root cause identified: agent sometimes writes to `output/` in the repo instead of `.fluid/working/` — skill instructions are ambiguous
- Ensure `.fluid/working/` directory exists at canvas startup (create if missing)
- No manual refresh button — auto-discovery must be bulletproof
- Watcher must handle the directory not existing initially and pick it up when created

### Skill path hardening
- Systematic audit needed of all path references across skills (fluid-social, fluid-design-os, fluid-one-pager, fluid-theme-section)
- Primary issue: output path ambiguity causing agent to write to wrong directory
- Skills must unambiguously direct output to `.fluid/working/{sessionId}/` when canvas is active
- Audit should cover: output paths, brand doc references, asset references, template paths

### Claude's Discretion
- Exact star icon design and placement
- How to handle rejected/final status controls (could be dropdown, right-click menu, etc.)
- File watcher implementation details (chokidar config, debounce timing)
- How to structure the path audit and prioritize fixes
- Template customization form design (which fields, layout)
- How to render the streaming JSON as chat messages in the sidebar
- Template gallery layout and navigation

</decisions>

<specifics>
## Specific Ideas

- "I just think that if we're powering this thing by headless claude CLI, we should ditch the idea of initiating it from claude desktop and embedding it in claude desktop entirely. Instead, we should just instigate the initial generation from the UI."
- "Prompt field/agent conversation is always there, maybe as a left sidebar, but in the main pane, there are template options to start from."
- User wants the sidebar to show the agent's actual thinking/progress — "chat-like stream" — not just a spinner or status dots
- "I'd like the user to be able to type their notes without picking a favorite. After all, that's just supposed to be general notes on that variation, right?"
- "We need to make sure we're saving the generated output to that folder rather than the output folder in the repo."
- Agent writing to `output/` instead of `.fluid/working/` was the specific file watcher failure observed

</specifics>
